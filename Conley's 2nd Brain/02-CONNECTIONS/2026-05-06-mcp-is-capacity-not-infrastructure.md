# The MCP Question Is Framed Wrong — It's Not Infrastructure, It's Capacity

**Bridge:** The question of whether to build the second brain MCP server (Phase I revenue work vs. Phase II infrastructure) is answered by the observation that every cold-start session drains the 15-hour cap re-orienting from scratch — meaning MCP integration isn't a build decision, it's a capacity reclamation decision.

**Potential hook:** "I've been asking the wrong question about whether to build the MCP server. The question wasn't 'does it generate revenue?' or 'is it Phase I work?' The question was simpler. Every session I start without it, I spend the first 20 minutes re-explaining what the wiki already knows. At 15 hours a week, that's not friction — that's a meaningful fraction of total capacity, burning on orientation. MCP isn't infrastructure spend. It's the cheapest capacity expansion available — no more hours, just fewer wasted ones."

**Why this matters:** The mcp-server-prioritization question frames the decision as Phase I (costs from the 15-hour weekly cap) versus Phase II (doesn't generate direct revenue). Neither frame is right. The wiki-as-second-context-layer observation shows that cold-start orientation is already costing time from the 15-hour cap — the MCP build cost is a one-time expenditure to eliminate a recurring weekly tax. The right comparison isn't "does MCP revenue justify the build?" but "how many orientation sessions equal the build cost, and what does that math say?" At 15 hours/week, eliminating even 30 minutes of weekly cold-start waste means MCP pays for itself in weeks, not quarters. The non-obvious reframe: building the MCP server isn't spending from the 15-hour cap — it's the only way to reclaim time that's already being spent without return. The Phase I/Phase II binary misses this entirely.
